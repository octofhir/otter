## Phase 2 step 2.7 — string-callsite migration plan

Scope: every site that today reads or writes a string via legacy
`HeapValue::String` / `Handle(u32)` / `JsString` must learn to also
handle `Value::TAG_PTR_STRING` / `GcRef<JsStringGc>`. Files touched
(63 total per `grep -l 'JsString\|HeapValue::String\|alloc_string'`):

* `intrinsics/string_class.rs` — every `String.prototype.*` method
* `intrinsics/regexp_class.rs` — capture extraction + replace + match
* `intrinsics/json.rs` — string serialisation + parser output
* `intrinsics/object_class.rs` — `Object.keys` / property name materialisation
* `intrinsics/error_class.rs` — message / stack rendering
* `intrinsics/symbol_class.rs` — `Symbol.prototype.description`
* `intrinsics/number_class.rs`, `bigint_class.rs`, `date_class.rs`,
  `array_class.rs`, `typedarray_class.rs`, `eval.rs`,
  `weakmap_weakset_class.rs`, `atomics.rs`, `reflect.rs`,
  `timer_globals.rs` — small leaks of the legacy path
* `interpreter/dispatch.rs` — bytecode opcodes that read/write string
  values: ToString conversion, concat, equality, etc.
* `interpreter/runtime_state/coercion.rs` — `js_to_string`,
  `to_property_key`, abstract relational comparison
* `interpreter/runtime_state/{mod,proxy,eval,alloc}.rs` — assorted
* `source_compiler/{mod,identifiers}.rs` — string literals in
  generated bytecode
* `value.rs`, `js_string.rs`, `string.rs`, `object.rs` — legacy struct
  homes; `object.rs` keeps `HeapValue::String` until step 2.8

Migration order (smallest blast radius first):

1. **Coercion layer** — `js_to_string`, `to_property_key`,
   `string_indexof_as_value`, etc. Once these understand
   `TAG_PTR_STRING`, every reader downstream gets it for free.
2. **`String.prototype.*`** — receiver coercion (`thisStringValue`)
   + return values. Each method is independent so progress is visible.
3. **`String` constructor / global** — `String(x)`, `String.fromCharCode`.
4. **JSON / RegExp** — large modules; landing each is a sub-phase.
5. **Source compiler** — string literals: `LoadConstant` ops emit
   `TAG_PTR_STRING` values directly.
6. **Bytecode dispatch** — opcodes that synthesize strings (Concat,
   ToString, NewError) dispatch through the new path.
7. **Interpreter mod** — property-key materialisation, error
   constructor, etc.
8. **Identifier interning** — replace the per-runtime `StringTable`
   so every interned identifier is a `GcRef<JsStringGc>`.
9. **Object dispatch** — `HeapValue::String` becomes a dead variant
   only used by tests that haven't migrated; step 2.8 deletes it.

Each numbered item lands as its own commit and run of
`cargo test -p otter-vm -p otter-runtime -p otter-gc --lib` plus a
focused `bash scripts/test262-safe.sh built-ins/String` sweep when
appropriate.

The same template applies to every subsequent variant track: the
allocation bridge already lives in `RuntimeState`, and adding a new
per-type GC struct only requires `register_all` to install the trace
function. Each variant's call-site sweep is its own multi-session
sub-phase.

**Strategy:** B — full V8-class production-grade rewrite. Per-variant
GC-managed structs (no `Box<dyn TypeErasedObject>` wrapper, no unified
`HeapValue` enum on the heap), raw `GcRef<T>` references replace
`Handle(u32)` on the entire VM call surface, page-based bump allocation,
write barriers on every heap-pointer store, semispace scavenger,
incremental marking, eventual concurrent marking. No compromises.

**Time budget:** unbounded. Each phase lands as a coherent monotonic
refactor with all tests green.

This plan retires the naive `TypedHeap` (`crates/otter-gc/src/typed.rs`) and
moves every VM allocation onto the page-based `GcHeap`
(`crates/otter-gc/src/heap.rs`) that was already designed but never wired up.

We do NOT use a feature flag. Each phase lands as a coherent refactor that
leaves the workspace green: `cargo build`, `cargo clippy`, full lib tests, and
the test262 sweep on the same subdir as the previous phase must pass before
the next phase starts. A phase is "closed" only when its acceptance bullets
are all checked here.

## Architectural diagnosis (verified Phase 0, 2026-04-26)

| Layer | Reality |
|---|---|
| `TypedHeap` (`typed.rs`) | Active path. `Vec<Option<Slot<Box<dyn TypeErasedObject>>>>`, naive STW mark-sweep with `Vec<bool>` marks parallel to slots. Pause @ 1 GB heap is 10–100 ms. No generational, no incremental, no write barriers. |
| `GcHeap` (`heap.rs`) | Dead code. `alloc_young` / `alloc_old` are bump-pointer (~10 cycles), `enforce_heap_limit` integrated. Zero call sites from `otter-vm`. |
| `scavenger.rs::scavenge` | Dead code. Generational copy-collect implemented. Zero call sites. |
| `marking.rs::drain_with_budget` | Dead code. Incremental marking + budget. Zero call sites. |
| `barrier.rs::WriteBarrier::record` | Dead code. Combined Yuasa SATB + Dijkstra insertion + generational. Zero call sites. |
| `header.rs::GcHeader` | Live (used only by `GcHeap` internals). 8B with type tag, mark color, young/old. AtomicU8 prepared for concurrent marking. |

The `lib.rs` doc comment ("Page-based generational tracing GC … incremental
marking … write barriers") is aspirational. Bringing it to truth is the
single biggest architectural bet in `PRODUCTION_READINESS_PLAN.md` §3.2.

## Phase 2 — `GcRef<T>` infrastructure + first migrated variant (`JsString`)

Phase 2 introduces V8-style raw-pointer references and migrates the
**first** `HeapValue` variant onto a per-type GC-managed struct. We pick
`JsString` as the pilot: it's a leaf type with no child references, so
the trace function is trivial and we exercise the full path
(allocation in pages, header type tag, page walk, mark/sweep, root
scanning, write barrier seam) without inheriting children-tracing
complexity.

Architecture deliverables:

1. **`GcRef<T>` smart pointer** (`crates/otter-gc/src/gc_ref.rs`):
   - `#[repr(transparent)] struct GcRef<T> { ptr: NonNull<GcHeader>, _phantom: PhantomData<*const T> }`
   - `Copy`, 8 bytes, no reference counting.
   - `impl<T> GcRef<T> { fn header(&self) -> &GcHeader; fn payload(&self) -> &T; }`
   - All access goes through `&Isolate` so the borrow checker prevents
     dangling refs across GC.

2. **`HandleScope` + `Local<T>`** (`crates/otter-gc/src/handle.rs` —
   already exists, extend with V8-style `Local<T>: Copy`):
   - `Local<T>` wraps `*const GcHeader` rooted in the `HandleStack`.
   - `HandleScope` is the RAII guard. Every native entrypoint that
     allocates wraps body in a `HandleScope`.
   - `enter_scope() / exit_scope()` already in `GcHeap`. We add
     ergonomic `Isolate::scope(|hs| { ... })` and per-call mappers.

3. **Per-type GC struct: `JsStringGc`** (`crates/otter-vm/src/string_gc.rs`):
   - `#[repr(C)] struct JsStringGc { header: GcHeader, len_units: u32, repr_tag: u8, padding: ..., units: [u16; 0] }`
     — flexible-array-member style; allocator picks size based on
     length and repr (Latin-1 / UTF-16 / cons / sliced / thin).
   - Single `TYPE_TAG_STRING = 1` registered with a `TraceFn` that
     visits child slots only for `Cons` / `Sliced` / `Thin` reprs
     (which carry `GcRef<JsStringGc>` children).
   - Every `JsString::intern / from_utf16 / new_gc / concat / slice /
     flatten / hash / eq / cmp` rewritten to operate on `GcRef<JsStringGc>`.

4. **`Isolate` (the single-threaded GC owner)**
   (`crates/otter-vm/src/isolate.rs` — already exists, augment):
   - Owns `GcHeap`, `string_table` (interner keyed on hash + payload),
     and the registered `TraceTable`.
   - Provides `alloc_string_latin1(&[u8]) -> Local<JsStringGc>` etc.
   - The thread-local `THREAD_STRING_TABLE` pattern stays but its
     entries are now `GcRef<JsStringGc>` (still weakly-referenced —
     the GC's intern-table sweep hook prunes dead entries at end of
     each cycle).

5. **NaN-box value tagging** (`crates/otter-vm/src/value.rs`):
   - `TAG_PTR_STRING` already exists. Its payload changes from
     `Handle(u32)` to `GcRef<JsStringGc>` (still 8 bytes). The bit
     layout has space for a 48-bit pointer.
   - `Value::as_string() -> Option<GcRef<JsStringGc>>` gets the new
     return type.

6. **All 50+ VM call sites that touch strings** rewrite to:
   - Take / return `GcRef<JsStringGc>` instead of `Handle(u32)`.
   - JIT helpers — `intrinsics/string_class.rs`, `js_string.rs`,
     `intrinsics/regexp_class.rs`, source-compiler string-literal
     encoder, `interpreter/runtime_state/coercion.rs`, every
     `String.prototype.*` method.

7. **JIT contract update** (`crates/otter-jit`):
   - `slots_ptr()` semantics change. `PagedHeap` does not have a slot
     table for strings any more — strings are accessed via `GcRef`
     pointers directly. JIT runtime helpers (`tier_up_hook.rs`,
     `deopt/resume.rs`, `code_memory.rs`) updated to dereference
     `*mut GcHeader` directly.
   - The slot-table contract still holds for non-migrated variants
     this phase. Once all variants migrate, the slot-table goes away.

8. **Co-existence with `TypedHeap`**:
   - `TypedHeap` keeps storing every other `HeapValue` variant.
   - `Isolate` owns BOTH `GcHeap` (for migrated types) and
     `TypedHeap` (for unmigrated types). Each `HeapValue::String`
     value moves out of the typed heap into `GcHeap`-backed
     `JsStringGc`. The `HeapValue::String` variant is removed.
   - This co-existence is dropped in Phase 3+ as more variants migrate.

Migration sequence within Phase 2:
1. Land `GcRef<T>`, `Local<T>`, `Isolate::scope` + tests.
2. Land `JsStringGc` per-type struct + trace function + intern table.
3. Land the new alloc / get / equality / hash API.
4. Switch `Value` string tagging to `GcRef<JsStringGc>`.
5. Sweep through every call site; remove `HeapValue::String`.
6. Update JIT runtime helpers.
7. Final lib + test262 sweep.

Tests: every existing `c2_string_*`, `c9_*`, `c10_*`, `c13_*` regression
must keep passing. Add new `c2b_gcref_*` tests for the GcRef-side API.

Acceptance: `JsString` lives in `GcHeap` pages; mark / sweep walks
strings through type tag dispatch; intern table prunes dead strings
via GC hook; all lib tests + Array/Object/String test262 sweeps at
parity.

## Phase 2-rest — migrate every other `HeapValue` variant onto per-type GC structs

After `JsString` proves the architecture, every remaining `HeapValue`
variant migrates to a per-type GC-managed struct. Each variant is its
own landing — tests must stay green between landings.

Order picked to land easiest-to-hardest, with leaf types first so the
write-barrier surface stays minimal until Phase 3:

| Variant | Notes / children |
|---|---|
| `BigInt` | Leaf (i64 inline + heap limbs `Vec<u64>`) |
| `UpvalueCell` | One `Value` field |
| `String iterator / Map iterator / Set iterator / Array iterator / Property iterator` | Hold one or two `GcRef`s |
| `Array` | `Vec<PropertyValue>` of values; sparse map |
| `Object` (the big one — shape + values vec) | Property values are `Value`s; shape is `Rc<Shape>` (not GC-managed yet — covered in Phase 2-shape) |
| `Closure` | Captures + function-bytecode `Rc<Function>` + `home_object` + `bound_this` |
| `BoundFunction` | Target + thisArg + boundArgs |
| `NativeObject / HostFunction` | Native payload via type-erased `Box<dyn>` — keep boxed payload on the GC heap as a separate `Pinned` allocation |
| `Map / Set` | Internal `IndexMap` / `IndexSet` of `(Value, Value)` |
| `WeakMap / WeakSet / WeakRef` | Use the existing ephemeron table, but with `GcRef` keys |
| `FinalizationRegistry` | Registration list + cleanup callback |
| `Promise + capability fns + thunks + finally fns + combinator-elements` | Several |
| `Generator + AsyncGenerator` | Saved registers `Box<[RegisterValue]>` — every register is a `Value` (now containing `GcRef`s) |
| `ArrayBuffer / SharedArrayBuffer / DataView / TypedArray` | Backing-store buffer; `SharedArrayBuffer` keeps `Arc<[u8]>` for cross-isolate sharing — **not** GC-moved |
| `RegExp` | Compiled pattern cache (`OnceCell<regress::Regex>`) — non-GC payload |
| `Proxy` | target + handler |
| `ErrorStackFrames` | Frame array |

Per-variant sub-phase:
1. Define `<Variant>Gc` struct with `#[repr(C)]` header + payload.
2. Register a unique type tag + `TraceFn`.
3. Add `Isolate::alloc_<variant>` constructor.
4. Convert every `HeapValue::<Variant>` callsite to `GcRef<<Variant>Gc>`.
5. Remove the `HeapValue::<Variant>` variant.
6. Run lib tests + appropriate test262 subdir.

Big-ticket items deserve their own dedicated sub-phases:

### Phase 2-shape — Move `Shape` onto the GC heap

`Shape` is currently `Rc<Shape>` (per "Phase 2.2 Arc→Rc landing").
Migrating to `GcRef<ShapeGc>` removes the refcount overhead and
integrates shape transitions into GC reachability (so dead shapes
get freed automatically when no object holds them). This is a
non-trivial sub-phase because:
- `transitions: RefCell<HashMap<(SymbolId, PropertyKey), Rc<Shape>>>`
  becomes a GC-owned mapping (every transition link is a `GcRef`
  edge that must be barrier-protected).
- Shape's `parent` chain becomes a `GcRef` chain.
- The proto-lookup cache (already RefCell-wrapped) must be traced.

### Phase 2-roots — Audit and centralize the root set

Before Phase 3 turns on barriers and Phase 4 turns on incremental
marking, every root must be discoverable. We enumerate:

- `Activation::registers` (`Vec<RegisterValue>` per call frame)
- `Activation::open_upvalues` map
- `accumulator` + `secondary_result`
- `closure_handle` + `construct_new_target`
- `pending_exception`
- `RuntimeState::current_module`
- `RuntimeState::frame_info_stack`
- `RuntimeState::pending_uncaught_throw`
- `RuntimeState::native_callee_stack`
- `RuntimeState::dynamic_import_*` thread-local
- All intrinsic root handles (`Intrinsics::install_on_global`)
- `proto_lookup_cache` (RefCell — already iterable)
- Generator `saved_registers` (per-generator)
- Microtask / event-loop queues

Each root site gains a `GcTraceable::trace(visitor)` impl. The
`Isolate::collect()` entrypoint walks every category, building the
root list passed to `GcHeap::collect_full / scavenge`.

## Hard invariants (violation = use-after-free)

1. Every heap-pointer write in a heap-stored cell goes through the
   barrier. Sanitizer in dev is non-negotiable for Phase 3.
2. No object compaction in Phases 1–5. `Handle(u32)` ↔ pointer mapping
   is stable for the lifetime of the heap. Compaction is post-Phase 6
   work.
3. GC safepoints are `RuntimeState::poll_back_edge` and the explicit
   allocator boundary. No GC from arbitrary instructions.
4. No `unsafe` outside `otter-gc`, `otter-jit`, `otter-modules/ffi.rs`.
   `// SAFETY:` comments mandatory in those crates.
5. Roots: `Activation::registers`, `open_upvalues`, `accumulator`,
   `secondary_result`, `closure_handle`, `construct_new_target`,
   `pending_exception`, plus `RuntimeState::current_module`,
   `frame_info_stack`, `pending_uncaught_throw`, `native_callee_stack`,
   `dynamic_import_*`, plus all intrinsic root handles. Phase 2's
   `collect` impl must scan all of these.
6. Shape transitions map: `(parent_shape, name) → child_shape` is
   `u64`-keyed; safe. If we ever key it on heap objects, it must be
   traced.
7. `proto_lookup_cache` holds `ObjectHandle` values. Trace it from
   `RuntimeState`-level roots (already RefCell-wrapped).
8. Generator `saved_registers` (`Box<[RegisterValue]>`) traced by
   `JsGenerator::trace`. Confirm under new backend.
